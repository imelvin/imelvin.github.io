java.concurrent包有两个子包：locks和atomic，我们先介绍子包的类。本文代码基于JDK 1.7源码。

java.util.concurrent.locks包

Lock接口

在Java 1.5以前，要给一个代码块或者方法加锁，只能通过synchronized关键字。自从1.5开始，Java提供了Lock接口和一系列实现，可以实现更复杂的锁机制。

Condition接口

Condition的对象是由Lock对象创建出来，Condition接口的方法类似于Object类上wait、signal、signalAll方法。Lock与Condition的关系类似于synchronized锁定的Object与Object类上wait、signal、signalAll方法之间的关系。Condition一定要在对应的Lock加锁以后才能使用。

ReadWriteLock接口

读写锁接口

LockSupport类

该类为创建锁和其他同步类提供了阻塞原语。LockSupport只是sun.misc.Unsafe的一个简单封装。该类提供了阻塞线程、唤醒线程的一系列方法，Lock和Condition的实现很多都需要使用到该类。

AbstractQueuedSynchronizer、AbstractQueuedLongSynchronizer和AbstractOwnableSynchronizer类

我们很少会直接使用LockSupport去构建自己的同步器，而是使用AQS也就是AbstractQueuedSynchronizer去构建。比如Semaphore、CountDownLatch等同步器就是在AbstractQueuedSynchronizer构建。在AbstractQueuedSynchronizer基础上构建自己的同步器会大大降低复杂度。

AbstractQueuedSynchronizer内部通过一个整形变量来表示状态，而AbstractQueuedLongSynchronizer的区别是该状态用的是长整形。

ReentrantLock类

该类应该是用的最多的锁，用来替换synchronized关键字。该类是Lock接口的一个实现，内部使用AbstractQueuedSynchronizer控制状态。

ReentrantReadWriteLock类

该类是读写锁接口的一个实现，可以分别获取读锁和写锁。同一时间可以有多个读线程，但是最多只能有一个写线程，而且写线程会阻塞读线程。很明显，这种锁适用于读操作远多于写操作的场景。

java.util.concurrent.atomic包

该包是一些使用CAS封装的基础数据类型，这些类都提供了CAS操作。

AtomicBoolean类

原子布尔型

AtomicInteger类

原子整型

AtomicLong类

原子长整型

AtomicReference类

原子引用类型

AtomicIntegerArray类

原子的整型数组

AtomicLongArray类

原子的长整型数组

AtomicReferenceArray类

原子的引用数组

AtomicIntegerFieldUpdater类

该类可以为某个类的volatile int字段提供原子更新器，提供CAS操作

AtomicLongFieldUpdater类

该类可以为某个类的volatile long字段提供原子更新器，提供CAS操作

AtomicReferenceFieldUpdater类

该类可以为某个类的volatile的引用字段提供原子更新器，提供CAS操作

AtomicMarkableReference类

原子的引用类型，但是带一个boolean的mark字段，mark字段与引用捆绑在一起提供原子操作。

AtomicStampedReference类

与AtomicMarkableReference类似，只是mark字段换为了一个int类型的stamp字段。

java.concurrent包

该包提供了一系列常用的同步器和一些高性能的容器，还有线程池，首先介绍这个包下的几个接口

接口

BlockingQueue接口

BlockingQueue集成了Queue接口，首先它是一个队列，然后它是一个阻塞队列。阻塞队列在生产者-消费者模型可以很好的降低实现复杂度。

BlockingDeque接口

BlockingDeque是一个双端阻塞队列。

TransferQueue接口

该接口扩展了BlockingQueue，但是该接口还提供了transfer方法，在添加元素时可以让生产者等待直到有消费者取走该元素。

Callable接口

该接口可看做Runnable接口的一个扩展，但是有返回值

CompletionService接口

类似于线程池，但是这个接口的实现都会返回Future对象。

ConcurrentMap接口

该接口扩展了Map接口，新增了putIfAbsent、remove、replace这些原子操作接口

Executor接口

该接口代表一个执行器，执行一个Runnable对象，但是并不限制用同步实现还是异步实现。

ExecutorService接口

该接口扩展了Executor接口，代表一个执行器服务，提供了更多关于ExecutorService生命周期管理的方法。一个典型实现就是ThreadPoolExecutor。

Future接口

该接口代表了一个异步任务的执行结果。

RunnableFuture接口

RunnableFuture接口将Runnable接口与Future接口结合起来，Future代表着run的结果

下面介绍一些常用的实现类

并发容器

ArrayBlockingQueue类

基于数组实现的一个固定大小FIFO的阻塞队列，用以实现有界缓存是一个很好的选择。

LinkedBlockingQueue类

基于链表实现的一个FIFO的阻塞队列，也可以指定容量，不指定的话，队列容量为Integer.MAX_VALUE。

PriorityBlockingQueue类

带优先级的阻塞队列，依赖于Comparable接口方法。底层数据结构是一个存储在数组上的平衡二叉树。

DelayQueue类

一个接受Delayed对象的阻塞队列，一个元素只有延迟一定时间后才能被消费者获取。

LinkedTransferQueue类

这是TransferQueue的一个实现，通过链表实现

ConcurrentLinkedQueue类

这是一个Queue而不是BlockingQueue，基于链表实现。采用CAS实现，没有加锁，所以多线程下效率很高。

ConcurrentLinkedDeque类

类似ConcurrentLinkedQueue，区别是这是一个双端队列。

ConcurrentHashMap类

优化过的线程安全的HashMap，采用了锁分段技术，而且读的时候不会加锁，并发性能好。

CopyOnWriteArrayList类

使用了CopyOnWrite技术实现的ArrayList，读时不用加锁，所以在读操作远多于写操作的并发场景应该使用该类。

CopyOnWriteArraySet类

该集合类基于CopyOnWriteArrayList实现

同步器

Semaphore类

信号量，基于QAS实现，类似于锁，但是可以控制进入临界区的线程数量

CountDownLatch类

闭锁，基于AQS实现。可以让一个或多个线程等待直到一些特定的事件发生。

CyclicBarrier类

栅栏，基于Condition实现。让一定数量的线程互相等待到达一个共同的栅栏位置。

并行框架

ThreadPoolExecutor类

这是用的最多的一个线程池类。

ForkJoinPool类

这是从1.7开始提供的Fork/Join框架，这是一个并行计算框架，典型的应用应该是将递归算法改为并行计算，将计算拆分成多个计算单元。

ScheduledThreadPoolExecutor类

类似于ThreadPoolExecutor，但是可以定时执行。

ExecutorCompletionService类

CompletionService的实现

总结

在多线程编程时，应该根据场景尽量使用concurrent包中提供的工具，可以使程序的安全性和性能得到保证。
